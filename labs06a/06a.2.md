# Lab 6a.2 – Deploying and Hardening the Webserver

## Learning Objectives

In this section you will:

- Deploy a simple NGINX webserver.
- Observe how Pod Security Standards affect workload creation.
- Troubleshoot Deployment, ReplicaSet and Pod failures.
- Add container security settings.
- Add CPU and memory requests and limits.
- Replace a root-based image with an unprivileged image.
- Adjust the replica count to comply with a ResourceQuota.
- Expose the final deployment using a ClusterIP Service.

---

## Why are we doing this?

In the previous section you prepared the namespace with several governance controls:

- a **Restricted Pod Security Standard**
- a **ResourceQuota**
- **NetworkPolicies**

In this section you will deploy a workload into that controlled namespace.

The important point is that the initial application manifest is not necessarily “wrong”. It is simply not compliant with the policies enforced in this environment.

This is a common real-world situation: platform teams define guardrails, and application teams must adjust their manifests to satisfy them.

---

## 1. Generate the initial Deployment manifest

Generate a starter Deployment manifest:

```bash
kubectl create deployment webserver \
  --replicas=10 \
  --image=nginx:alpine \
  --port=80 \
  --dry-run=client -o yaml > deploy.yaml
```

Open the file:

```bash
nano deploy.yaml
```

You should see a Deployment similar to this:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: webserver
  name: webserver
spec:
  replicas: 10
  selector:
    matchLabels:
      app: webserver
  template:
    metadata:
      labels:
        app: webserver
    spec:
      containers:
      - image: nginx:alpine
        name: nginx
        ports:
        - containerPort: 80
        resources: {}
```

This is a valid Kubernetes Deployment, but it does not yet meet the policies enforced in the `webserver` namespace.

---

## 2. Apply the Deployment

Apply the manifest:

```bash
kubectl apply -n webserver -f deploy.yaml
```

You should see a warning or failure related to the **Restricted Pod Security Standard**.

Check whether any Pods were created:

```bash
kubectl get pods -n webserver
```

Expected result:

```text
No resources found in webserver namespace.
```

The Deployment object may exist, but the Pods are not admitted because they do not meet the namespace security requirements.

---

## 3. Review the Deployment

Describe the Deployment:

```bash
kubectl describe deployment webserver -n webserver
```

Look carefully at the **Events** section.

The Deployment controller is trying to create Pods, but the Pods are being rejected because the container specification does not satisfy the Restricted policy.

At this stage, the problem is not the image itself. The first problem is that the required security settings are missing from the manifest.

---

## 4. Add a container securityContext

Edit `deploy.yaml`.

Add the following `securityContext` inside the container definition.

Be careful with indentation. The `securityContext` belongs under the container, at the same level as `image`, `name`, `ports` and `resources`.

```yaml
securityContext:
  runAsNonRoot: true
  allowPrivilegeEscalation: false
  capabilities:
    drop:
    - ALL
  seccompProfile:
    type: RuntimeDefault
```

Your container should now look like this:

```yaml
containers:
- image: nginx:alpine
  name: nginx
  securityContext:
    runAsNonRoot: true
    allowPrivilegeEscalation: false
    capabilities:
      drop:
      - ALL
    seccompProfile:
      type: RuntimeDefault
  ports:
  - containerPort: 80
  resources: {}
```

Apply the manifest again:

```bash
kubectl apply -n webserver -f deploy.yaml
```

Check the Pods:

```bash
kubectl get pods -n webserver
```

You may still see no running Pods.

That is expected. You have solved the first policy problem, but another one is now visible.

---

## 5. Investigate the ReplicaSet

A Deployment creates a ReplicaSet, and the ReplicaSet creates Pods.

If the Deployment exists but Pods are not appearing, the next place to investigate is the ReplicaSet.

List ReplicaSets:

```bash
kubectl get rs -n webserver
```

Describe the ReplicaSet:

```bash
kubectl describe rs -n webserver
```

You should see events indicating that Pod creation is blocked by the namespace `ResourceQuota`.

The quota requires CPU and memory values to be declared. Because the container still has:

```yaml
resources: {}
```

the ReplicaSet cannot create Pods.

---

## 6. Add resource requests and limits

Edit `deploy.yaml`.

Replace:

```yaml
resources: {}
```

with:

```yaml
resources:
  requests:
    cpu: 100m
    memory: 100Mi
  limits:
    cpu: 100m
    memory: 100Mi
```

Your container should now look like this:

```yaml
containers:
- image: nginx:alpine
  name: nginx
  securityContext:
    runAsNonRoot: true
    allowPrivilegeEscalation: false
    capabilities:
      drop:
      - ALL
    seccompProfile:
      type: RuntimeDefault
  ports:
  - containerPort: 80
  resources:
    requests:
      cpu: 100m
      memory: 100Mi
    limits:
      cpu: 100m
      memory: 100Mi
```

Apply the manifest again:

```bash
kubectl apply -n webserver -f deploy.yaml
```

Check the Pods:

```bash
kubectl get pods -n webserver
```

You should now see some Pods, but they may not be running successfully.

Example:

```text
NAME                         READY   STATUS                       RESTARTS   AGE
webserver-xxxxxxxxxx-xxxxx   0/1     CreateContainerConfigError   0          20s
```

This is progress. The Pods are now being created, but the containers cannot start.

---

## 7. Investigate the Pod failure

Describe one of the failed Pods:

```bash
kubectl describe pod <pod-name> -n webserver
```

Look at the **Events** section.

The issue is now related to the image and the `runAsNonRoot` setting.

The standard `nginx:alpine` image expects to run as root. You have explicitly told Kubernetes that the container must not run as root:

```yaml
runAsNonRoot: true
```

Kubernetes therefore refuses to start the container.

This is an important lesson: Kubernetes security settings and container image design must agree with each other.

---

## 8. Switch to an unprivileged NGINX image

Edit `deploy.yaml`.

Replace the image:

```yaml
image: nginx:alpine
```

with:

```yaml
image: nginxinc/nginx-unprivileged
```

The unprivileged NGINX image is designed to run as a non-root user.

Because unprivileged containers cannot bind to privileged ports such as port 80, also change the container port from `80` to `8080`:

```yaml
ports:
- containerPort: 8080
```

Your container should now look like this:

```yaml
containers:
- name: nginx
  image: nginxinc/nginx-unprivileged
  securityContext:
    runAsNonRoot: true
    allowPrivilegeEscalation: false
    capabilities:
      drop:
      - ALL
    seccompProfile:
      type: RuntimeDefault
  ports:
  - containerPort: 8080
  resources:
    requests:
      cpu: 100m
      memory: 100Mi
    limits:
      cpu: 100m
      memory: 100Mi
```

Apply the manifest again:

```bash
kubectl apply -n webserver -f deploy.yaml
```

Check the Pods:

```bash
kubectl get pods -n webserver
```

You should now see running Pods.

However, you may only see five Pods, even though the Deployment still asks for ten replicas.

---

## 9. Explain the replica mismatch

Check the Deployment:

```bash
kubectl get deployment webserver -n webserver
```

You may see something like:

```text
NAME        READY   UP-TO-DATE   AVAILABLE
webserver   5/10    5            5
```

This is not an application error. It is the `ResourceQuota` doing its job.

Earlier, the namespace quota limited the namespace to five Pods:

```bash
pods=5
```

The Deployment is requesting ten replicas, but the namespace is only permitted to run five Pods.

---

## 10. Reduce the replica count

Edit `deploy.yaml`.

Change:

```yaml
replicas: 10
```

to:

```yaml
replicas: 5
```

Apply the manifest:

```bash
kubectl apply -n webserver -f deploy.yaml
```

Verify the Deployment:

```bash
kubectl get deployment webserver -n webserver
kubectl get pods -n webserver
```

Expected result:

```text
NAME        READY   UP-TO-DATE   AVAILABLE
webserver   5/5     5            5
```

---

## 11. Expose the Deployment

Create a ClusterIP Service for the Deployment:

```bash
kubectl expose deployment webserver \
  -n webserver \
  --type=ClusterIP \
  --port=8080 \
  --target-port=8080
```

Verify:

```bash
kubectl get svc -n webserver
```

Expected result:

```text
NAME        TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)
webserver   ClusterIP   10.x.x.x        <none>        8080/TCP
```

---

## 12. Optional verification from inside the cluster

If you have a temporary test Pod available, you can test the Service from inside the cluster.

For example:

```bash
kubectl run curl-test \
  --rm -it \
  --image=curlimages/curl \
  --restart=Never \
  -- sh
```

From inside the Pod:

```bash
curl http://webserver.webserver.svc.cluster.local:8080
```

You should see the default NGINX welcome page.

Exit the temporary Pod:

```bash
exit
```

Depending on the NetworkPolicies currently in place, direct access may or may not be permitted from arbitrary namespaces. In the next section you will route traffic through the NGINX Ingress Controller.

---

## Final deploy.yaml

Your completed Deployment should look similar to this:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: webserver
  name: webserver
spec:
  replicas: 5
  selector:
    matchLabels:
      app: webserver
  template:
    metadata:
      labels:
        app: webserver
    spec:
      containers:
      - name: nginx
        image: nginxinc/nginx-unprivileged
        securityContext:
          runAsNonRoot: true
          allowPrivilegeEscalation: false
          capabilities:
            drop:
            - ALL
          seccompProfile:
            type: RuntimeDefault
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: 100m
            memory: 100Mi
          limits:
            cpu: 100m
            memory: 100Mi
```

---

## Summary

You have now deployed a webserver that satisfies the governance controls applied to the namespace.

You resolved several separate issues:

| Issue | Cause | Fix |
|---|---|---|
| Pods rejected by Pod Security | Missing security settings | Added container `securityContext` |
| Pods blocked by ResourceQuota | Missing CPU/memory requests and limits | Added `resources` stanza |
| Container failed to start | Image expected to run as root | Switched to `nginxinc/nginx-unprivileged` |
| Only five Pods created | Namespace quota allowed only five Pods | Reduced replicas from 10 to 5 |

The application is now running behind a ClusterIP Service. In Lab 6a.3 you will expose it through the NGINX Ingress Controller and troubleshoot cross-namespace routing.
