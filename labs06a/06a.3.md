# Lab 6a.3 – Configuring and Troubleshooting Ingress

## Learning Objectives

In this section you will:

- Create an Ingress resource.
- Understand why Ingress backends are normally resolved in the same namespace.
- Troubleshoot failed ingress routing.
- Use an `ExternalName` Service to bridge namespaces.
- Identify DNS as a dependency of `ExternalName`.
- Update a NetworkPolicy to allow DNS egress.
- Verify that the webserver is reachable through the ingress controller.

---

## Why are we doing this?

In the previous section you deployed a compliant webserver into the `webserver` namespace and exposed it using a ClusterIP Service.

In this section you will expose that application through the NGINX Ingress Controller running in the `ingress` namespace.

This deliberately introduces a common real-world problem: the ingress controller and the backend application are in different namespaces, and network policies are restricting traffic.

## 1. Create a starter Ingress manifest

Create an Ingress manifest in the `ingress` namespace:

```bash
kubectl create ingress new-ingress \
  -n ingress \
  --class=nginx \
  --rule="/*=webserver:8080" \
  --dry-run=client -o yaml > ingress.yaml
```

Review the generated file:

```bash
nano ingress.yaml
```

It should look similar to this:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: new-ingress
  namespace: ingress
spec:
  ingressClassName: nginx
  rules:
  - http:
      paths:
      - backend:
          service:
            name: webserver
            port:
              number: 8080
        path: /
        pathType: Prefix
```

Apply it:

```bash
kubectl apply -f ingress.yaml
```

Verify:

```bash
kubectl get ingress -n ingress
```

## 3. Confirm that the webserver is healthy

Before blaming Ingress, confirm that the webserver itself is running.

```bash
kubectl get pods -n webserver
kubectl get svc -n webserver
```

Check the webserver Pod logs:

```bash
kubectl logs -n webserver -l app=webserver
```

If the webserver Pods are healthy but the browser request does not reach them, the problem is probably in the routing path between the ingress controller and the backend Service.

---

## 4. Understand the namespace problem

The Ingress resource was created in the `ingress` namespace.

Its backend refers to a Service named:

```text
webserver
```

Ingress backends are resolved in the same namespace as the Ingress resource.

That means the Ingress Controller expects to find this Service:

```text
webserver.ingress.svc.cluster.local
```

But the real Service exists here:

```text
webserver.webserver.svc.cluster.local
```

So the Ingress configuration is pointing at a Service name that does not exist in the `ingress` namespace.

---

## 5. Create an ExternalName Service

To solve the namespace mismatch, create a Service named `webserver` in the `ingress` namespace.

This Service will point to the real Service in the `webserver` namespace.

Review the provided manifest:

```bash
nano ~/qakf-3day/solutions/lab6a/ename_svc.yaml
```

It should look like this:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: webserver
  namespace: ingress
spec:
  type: ExternalName
  externalName: webserver.webserver.svc.cluster.local
  ports:
  - port: 8080
    targetPort: 8080
```

Apply it:

```bash
kubectl apply -f ~/qakf-3day/solutions/lab6a/ename_svc.yaml
```

Verify:

```bash
kubectl get svc -n ingress
```

You should now see a Service named `webserver` in the `ingress` namespace.

Reload the browser.

The application may still not work.

That is also expected.

You have solved the namespace lookup problem, but introduced another dependency: DNS.

---

## 6. Understand the DNS problem

An `ExternalName` Service is effectively a DNS alias.

The ingress controller now has to resolve:

```text
webserver.ingress.svc.cluster.local
```

which points to:

```text
webserver.webserver.svc.cluster.local
```

To do that, the ingress controller must be able to query Kubernetes DNS.

Kubernetes DNS normally runs in the `kube-system` namespace as Pods labelled:

```yaml
k8s-app: kube-dns
```

However, the `ingress` namespace has an egress NetworkPolicy. That policy currently allows traffic from the ingress controller to the webserver namespace, but it does not allow DNS egress to kube-dns.

So the ingress controller cannot resolve the backend name.

---

## 7. Update the ingress NetworkPolicy

Open the ingress NetworkPolicy:

```bash
nano netpol_ingress.yaml
```

It should already allow egress traffic to the `webserver` namespace on TCP port 8080.

Add a second egress rule allowing DNS lookups to kube-dns in the `kube-system` namespace.

Use this completed version if required:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: ingress-netpol
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          app: webserver
    ports:
    - protocol: TCP
      port: 8080
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
```

Apply the updated policy:

```bash
kubectl apply -n ingress -f netpol_ingress.yaml
```

Verify:

```bash
kubectl describe networkpolicy ingress-netpol -n ingress
```


```bash

kubectl get svc webserver -n webserver -o yaml
```

## Summary

In this lab you exposed a locked-down application through an ingress controller while maintaining namespace and network separation.

You resolved three separate routing problems:

| Problem | Cause | Resolution |
|---|---|---|
| Ingress could not reach the backend | Backend Service was in a different namespace | Created an `ExternalName` Service in the ingress namespace |
| ExternalName still failed | DNS lookups were blocked by egress policy | Allowed egress to kube-dns on port 53 |
| Direct backend exposure avoided | Webserver namespace only allows traffic from ingress | Used NetworkPolicy to control access |

The completed design is:

```text
Browser
  |
  v
NodePort on ingress controller
  |
  v
NGINX Ingress Controller Pod
  |
  v
webserver Service in ingress namespace
  |
  v
ExternalName DNS alias
  |
  v
webserver Service in webserver namespace
  |
  v
webserver Pods on port 8080
```

This is a realistic example of how Kubernetes policy, DNS, Services and Ingress interact in a governed cluster.
